# Frontend Implementation Plan — Tech Lead Simulator (MVP)

## Context

The game's frontend is a **bare `create-vue` scaffold** (Vue 3.5, Vite 8, TypeScript 6) — only a placeholder `App.vue`, no router, store, HTTP layer, styles, or types. We have a contract-first `openapi.yaml` (single source of truth for BE↔FE) and a design sheet `design/third-variant.jpg` with ~14 numbered screen mockups. The backend has **no REST layer yet**, so the frontend must be buildable and playable independently.

This plan builds the full MVP game loop as a Vue SPA that runs today against mocked endpoints and later flips to the real backend with an env flag.

### Locked decisions (confirmed with user)
- **Scope:** full core loop **+ the AI analysis/roadmap screen** (`GET /ai-result`, no mockup — designed to match the flat style). Lobby & video-call are light transition screens.
- **Backend during dev:** **MSW mocks** built from the OpenAPI examples, with a `VITE_USE_MOCKS` toggle to hit the real `:8080` backend via a Vite proxy later.
- **Styling:** **plain CSS + design tokens** (CSS custom properties), scoped component styles. No CSS framework.
- **API types:** **generated from `openapi.yaml`** via `openapi-typescript`, consumed through **`openapi-fetch`** (typed, zero-runtime client) as the thin fetch wrapper.

> Per CLAUDE.md: these are recent major versions (Vite 8, Vue 3.5). Verify exact setup against Context7 during implementation. This document is the plan only — no implementation code is committed yet.

## Architecture

### Dependencies to add
- Runtime: `vue-router`, `pinia`, `openapi-fetch`
- Dev: `openapi-typescript`, `msw`
- MSW needs `npx msw init frontend/public/ --save` to drop the service-worker file into `public/`.

### Target `frontend/src/` layout
```
src/
├── main.ts                     # bootstrap: pinia, router, global CSS, conditional MSW start
├── App.vue                     # <RouterView/> + global PauseModal mount
├── router/index.ts             # routes + guard (redirect to /mode if no active interview)
├── api/
│   ├── schema.d.ts             # GENERATED by openapi-typescript (do not hand-edit)
│   └── client.ts               # createClient<paths>({ baseUrl }) from openapi-fetch
├── stores/
│   ├── interview.ts            # session + per-round game state (Pinia)
│   └── home.ts                 # home payload + player stats
├── mocks/
│   ├── browser.ts              # setupWorker(...handlers)
│   ├── handlers.ts             # one handler per operationId
│   └── fixtures.ts             # in-memory session state driving the mock game
├── composables/
│   └── useCountdown.ts         # question timer (timeLimitSeconds)
├── components/                 # shared UI (see below)
├── views/                      # one per screen (see routes)
└── assets/styles/
    ├── tokens.css              # CSS custom properties (design tokens)
    └── base.css                # reset + typography, imported once in main.ts
```

### Routes → screens → API
| Route | View | Design screen | API call |
|---|---|---|---|
| `/` | `HomeView` | 1 Main menu, 3 Stats | `getHomePage` |
| `/mode` | `ModeSelectView` | 2 Mode select | uses `home.modes` |
| `/interview/:id/lobby` | `LobbyView` | 4 Lobby, 5 Video call | (from `startInterview` result) |
| `/interview/:id/question` | `QuestionView` | 6 Question, 7 Answers, 10 Intermediate stats | `getQuestion`, `saveAnswer` |
| `/interview/:id/result` | `ResultView` | 13 Result, 14 Breakdown | `getInterviewResult` |
| `/interview/:id/offer` | `OfferView` | 15 Choose candidate, 16 Team | `getInterviewStatistic`, `offer` |
| `/interview/:id/summary` | `SummaryView` | 17 Final result | `getAiInterviewResult` (poll on 202) |
| overlay | `PauseModal` | 20 Pause | — (client-only) |

`startInterview` (`POST /interviews`) is triggered from `ModeSelectView`, then routes to `/interview/:id/lobby`.

### Pinia `interview` store — the game's spine
State: `session` (interviewId, mode, difficulty, totalQuestions, candidates), `currentQuestion`, `lastFeedback` (AnswerResult), running totals (`correctCount`, `currentStreak`, `totalPoints`, `answeredCount`), `statistic`, `offerResult`, `result`, `aiResult`.
Actions wrap each endpoint via `api/client.ts`: `start(context)`, `loadQuestion()`, `submitAnswer(answerId)`, `loadStatistic()`, `makeOffer(personId)`, `loadResult()`, `loadAiResult()` (retries while `status === PENDING` / HTTP 202).
Route guard: navigating into `/interview/:id/*` without a matching loaded session redirects to `/`.

### Shared components (`components/`)
`AppButton`, `AppCard`, `ProgressBar` (question index + score bars), `CandidateAvatar` / `CandidateCard` (avatar, name, role, strengths — reused across lobby, answers, offer, team), `AnswerOptionRow` (answer text + candidate avatar; green/red state after feedback), `StatTile` (games played / win rate / best result), `ScoreBadge`, `PauseModal`, `RoadmapItem`.

### Design tokens (`tokens.css`) — approximate from the mockup, tune to design
Primary blue `#2F6BFF` (buttons, progress), light app bg `#F5F6F8`, white cards, success `#22C55E`, error `#EF4444`, heading text `#1F2937`, muted `#6B7280`, card radius `~14px`, soft shadow, base spacing scale (4/8/12/16/24), system/`Inter`-ish font stack.

### API client & env config
- `.env.development`: `VITE_API_BASE=/api`, `VITE_USE_MOCKS=true`.
- `api/client.ts`: `createClient<paths>({ baseUrl: import.meta.env.VITE_API_BASE })`. All calls destructure `{ data, error }`; `error` matches the `Error` schema (`code`, `message`) and surfaces a toast/inline message.
- `vite.config.ts`: add dev proxy `/api → http://localhost:8080/api` (used when `VITE_USE_MOCKS=false`).

### MSW mock layer
- `main.ts` gates start on the flag before mounting:
  ```ts
  async function enableMocking() {
    if (!import.meta.env.VITE_USE_MOCKS) return
    const { worker } = await import('./mocks/browser')
    return worker.start({ onUnhandledRequest: 'bypass' })
  }
  enableMocking().then(() => app.mount('#app'))
  ```
- `handlers.ts`: one `http.*` handler per operationId returning `HttpResponse.json(...)` seeded from the OpenAPI `examples`. `fixtures.ts` keeps a tiny in-memory session map so the mock plays a real 10-round game: tracks answered index, picks a correct answer, accumulates score/streak, and serves statistic/result/ai-result consistently. Include a one-off 202 for `ai-result` then `READY` to exercise the polling path.

## Implementation phases

1. **Foundation** — install deps; `openapi-typescript ../openapi.yaml -o src/api/schema.d.ts` (add an `npm run gen:api` script); `api/client.ts`; env files + Vite proxy; `tokens.css` + `base.css` imported in `main.ts`; rename `index.html` title.
2. **App shell** — Pinia + router wired in `main.ts`; `App.vue` → `<RouterView/>`; route table + guard; empty view stubs so navigation compiles.
3. **Mock backend** — `msw init`; `handlers.ts` + `fixtures.ts` + `browser.ts`; conditional start. Milestone: endpoints respond in the network tab.
4. **Home + start flow** — `home` store, `HomeView` (menu + stat tiles), `ModeSelectView` → `startInterview` → `LobbyView`.
5. **Core game loop** — `interview` store; `QuestionView` with `AnswerOptionRow`s, timer composable, submit → inline correct/incorrect feedback → intermediate-stats panel → next; advance until `finished`.
6. **Results & offer** — `ResultView` (score + per-question breakdown), `OfferView` (statistic-driven candidate carousel → `offer` → team panel).
7. **AI summary** — `SummaryView`: final points + `getAiInterviewResult` with 202-polling, verdict, and roadmap list.
8. **Polish** — `PauseModal` (restart / settings / exit), loading & error states, empty/`404`/`409` handling, responsive check against the mockup, run `npm run lint` + `type-check`.

### Shared UI components: reuse guidance
Build `CandidateCard`/`CandidateAvatar` once in phase 4 (lobby) — they recur in answers (5), offer (6), team (6) and must not be re-implemented per screen. Same for `ProgressBar` (used for question progress and score bars).

## Verification (end-to-end)

- **Types generate cleanly:** `npm run gen:api` produces `src/api/schema.d.ts`; `npm run type-check` passes.
- **Build/lint:** `npm run build` (vue-tsc + vite) and `npm run lint` succeed.
- **Full playthrough against mocks** (`npm run dev`, `VITE_USE_MOCKS=true`): from Home → pick a mode → play all 10 rounds (verify feedback, streak, score, timer) → result + breakdown → statistic-driven offer → team → AI summary (confirm the 202→READY polling renders the roadmap). Confirm the guard: deep-linking `/interview/999/question` with no session redirects home.
- **Contract switch smoke test:** set `VITE_USE_MOCKS=false`; confirm requests hit `/api/...` through the Vite proxy (will 404 until backend exists — expected; proves the toggle works).
- No unit-test suite is set up in the scaffold; MVP verification is the manual playthrough above. (Optional later: add Vitest for the `interview` store's scoring logic.)

## Out of scope (MVP)
Auth/player accounts, persistence of `PlayerStats` (comes from `getHomePage` only), settings screen internals, i18n, animations beyond the confetti/celebration still, and real backend implementation.

## Open assumptions (flag if wrong)
- Mode select maps design's "Карьерный режим" → `CLASSIC`, "Быстрая игра" → `HARDCORE`; `difficulty` defaults to `MEDIUM` with no dedicated UI in MVP.
- Candidate avatar images (`avatarUrl`) are served as static assets; mocks reference `/assets/candidates/*` placeholders.
